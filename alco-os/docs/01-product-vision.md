# 01. プロダクトビジョン

## 一文で

**現場の観察とメモが、承認ひとつで業務文書・タスク・経営数字に変わる。**

## モジュール構成

```
ALCO OS
├─ Core          認証・権限・監査ログ・ファイル・AI実行ログ・タスク
├─ Voice Memo    音声メモ → TODO / 議事録 / 申請素材 / 日報（実装済MVP）
├─ Gibier        個体→解体→製品→在庫→受注→納品（既存システムを統合）
├─ Grants        補助金・要件整理・申請書・実績報告（実装済MVP）
├─ Nature Capital 調査地点・生物記録・証跡・レポート（実装済MVP）
├─ CRM           顧客・紹介者・面談・案件・次アクション（実装済MVP）
├─ Projects      ROKA改修・拠点整備・意思決定ログ（実装済MVP）
├─ HR            SOP・チェックリスト（シフト勤怠は既存Gibier側）
├─ Documents     社内Wiki・テンプレート・AI参照資料
└─ Dashboard     経営KPI・在庫・売上・案件・未処理タスク（実装済MVP）
```

## 中核となる設計思想（3つ）

### 1. ドラフト承認フロー（AIは勝手に業務データを書き換えない）

```
生入力（メモ・調査・要領）
  ↓ AI処理（ai_runs に記録）
generated_drafts（draft）
  ↓ 人間レビュー（承認センター /drafts）
業務テーブル反映（tasks / grant_documents / ...）
  ↓
audit_logs（監査ログ）
```

### 2. モジュール追加で育てる（アプリを分けない）

すべてのモジュールが organizations / profiles / tasks / files /
ai_runs / audit_logs / generated_drafts を共有する。
新機能は「新アプリ」ではなく「新モジュール」として追加する。

### 3. モデル非依存（AI adapter）

AI呼び出しは必ず `src/ai/model-router.ts` を経由する。
Fable / Opus / Sonnet / 他社モデルへの切り替えは環境変数と
model-router の変更だけで完了し、アプリ側は壊れない。

## 成功の定義

- 代表の音声メモが、その日のうちにタスク・記録・文書素材になる
- 補助金・レポートのドラフト作成時間が 1/5 になる（提出前レビューは必ず人間）
- 経営数字（売上・在庫・案件・未処理）が一画面で見える
- スタッフが増えてもSOP・チェックリストで品質が維持される
